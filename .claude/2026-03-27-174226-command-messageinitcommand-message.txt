⏺ A few concrete suggestions:

  1. .gitignore ignores *.png, but PNGs are checked in

  The .gitignore at line 211 has *.png, yet all the figure files (panel_a.png,
  D10.png, FI_comparing.png, etc.) are tracked in the repo. This works because
  they were already committed before the rule was added, but it's fragile — git 
  add will silently skip any new PNG you try to stage. Either remove *.png from
  .gitignore (intentionally tracking figures is fine for a paper repo) or
  document the workaround (git add -f).

  2. CI doesn't run the heavy simulation scripts

  The CI runs only plot.py scripts (which load pre-computed .npy/.npz files) but
   skips simulation.py scripts. That means a breaking change to simulation.py
  won't be caught. This is probably intentional given runtime, but worth a
  comment in the CI file to make the intent explicit — e.g., # simulation.py 
  scripts are excluded: they require HPC runtimes (hours).

  3. CI installs uv via curl | sh without pinning a version

  curl -LsSf https://astral.sh/uv/install.sh | sh

  This fetches the latest uv on each run, which could silently break. Pinning it
   is easy:

  uses: astral-sh/setup-uv@v5
  with:
    version: "0.6.x"

  4. The adapt_KM&G_dynamics/ directory name

  The & in the directory name requires escaping in every shell command
  (adapt_KM\&G_dynamics) and is the source of the awkward cd 
  ../adapt_KM\&G_dynamics in CI. Renaming to adapt_KMG_dynamics or
  adapt_dynamics would be cleaner, though it's a git history rewrite so only
  worth doing if you haven't publicized the current URL widely.

  5. The commented-out CI output check

  Lines 79–83 in ci.yml have a useful sanity check (test -f outputs/figure.png)
  but it's commented out. Since the figures are regenerated by the plot scripts,
   enabling those checks would make CI actually verify outputs were produced,
  not just that scripts didn't crash.

  6. Minor: arXiv badge URL mismatch in README

  The badge label says arXiv:2503.03126 but links to arXiv:2509.00219. One of
  those numbers is wrong.

❯ Can I move the CLAUDE.md file into a hidden directory?                        

⏺ No — Claude Code specifically looks for CLAUDE.md at the project root (and
  parent directories). If it's in a hidden directory, it won't be loaded
  automatically.

  If you want to keep the repo tidy, the supported alternatives are:
  - Add CLAUDE.md to .gitignore so it's present locally but not committed
  - Keep it at the root (it won't affect the paper repo's presentation on GitHub
   since it's just one extra file)
